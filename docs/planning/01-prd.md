## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

**종이 교적부의 한계:**
- 매주 일일이 손으로 작성해야 해서 10분 이상 소요
- 나중에 찾기 어려워서 학생 이탈 조짐을 놓치기 쉬움
- 엑셀로 관리하려니 복잡하고, 데이터가 여러 곳에 흩어져 있음
- 교사들이 행정 업무에 시간을 빼앗겨 사역의 본질인 '돌봄'에 집중하기 어려움

**비유:** 마치 옛날에 종이 일기장에 일일이 기록하던 것처럼, 매주 반복되는 수작업에 지쳐가는 상황입니다.

### 1.2 목표 상황 (To-Be)

**스마트 교적부의 이상:**
- 교사가 스마트폰으로 **3분 만에** 출석 체크 완료 (기존 10분 → 3분)
- 학생이 3주 이상 결석하면 자동으로 알림이 와서 '골든타임'을 놓치지 않음
- 모든 데이터가 한 곳에 모여 있어서 언제든지 쉽게 조회 가능
- 행정 시간이 줄어서 교사들이 학생 돌봄에 더 집중할 수 있음

**비유:** 마치 스마트폰으로 카드 결제하듯이, 터치 한 번으로 출석이 체크되고 모든 기록이 자동으로 정리되는 세상입니다.

### 1.3 왜 지금인가? (Why Now)

- **AI 개발 도구의 성숙:** Cursor, Windsurf 같은 AI 코딩 도구가 발전하여 1인 개발자도 엔터프라이즈급 시스템 구축 가능
- **클라우드 인프라의 접근성:** Supabase 같은 BaaS로 백엔드 구축 비용과 시간 대폭 절감
- **모바일 퍼스트 시대:** 교사들이 스마트폰을 일상적으로 사용하여 모바일 앱에 대한 거부감 감소
- **데이터 기반 목회의 필요성:** 학생 이탈을 조기에 발견하고 예방하는 것이 더 중요해짐

### 2.2 페르소나의 하루 (시나리오)

**주일 오전 9:00 - 출석 체크**
1. 스마트폰으로 앱 로그인
2. "유년부 > 3학년 > 사랑반" 선택
3. 학생 카드 목록 확인
4. 출석한 학생 카드 터치 → 즉시 체크 완료 (초록색 테두리)
5. 결석한 학생 카드 터치 → "아픔" 사유 선택
6. **3분 만에 출석 체크 완료!** (기존 10분 → 3분)
7. "오늘 출석 체크 완료! 🎉" 메시지 확인

**평일 저녁 8:00 - 심방 기록**
1. 앱 알림 확인: "김철수 학생이 3주째 결석 중입니다"
2. 전화 심방 후 앱에서 기록
3. 심방 타입 선택: "전화"
4. 내용 입력: "가정 문제로 힘들어함"
5. 기도 제목 입력: "가정 화목을 위해"
6. 타임라인에 자동 추가됨

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩/로그인)

- [ ] 교사로서, 이메일과 비밀번호로 간편하게 로그인할 수 있어야 한다. 왜냐하면 매주 사용할 것이므로 빠르게 접근해야 하기 때문이다.
- [ ] 교사로서, 로그인 상태가 유지되어야 한다. 왜냐하면 매주 다시 로그인하는 것이 번거롭기 때문이다.
- [ ] 교사로서, 첫 사용 시 3분짜리 튜토리얼을 볼 수 있어야 한다. 왜냐하면 디지털 기기에 익숙하지 않은 교사도 쉽게 사용할 수 있어야 하기 때문이다.

### FEAT-1: 스마트 출석 체크 (MVP)

- [ ] 바쁜 김교사로서, 스마트폰에서 학생 카드를 터치 한 번으로 출석/결석을 처리할 수 있어야 한다. 왜냐하면 출석 체크 시간을 10분에서 3분으로 단축하고 싶기 때문이다.
- [ ] 바쁜 김교사로서, 결석한 학생의 사유(아픔, 여행, 늦잠 등)를 쉽게 선택할 수 있어야 한다. 왜냐하면 나중에 통계를 내거나 심방할 때 참고하기 때문이다.
- [ ] 바쁜 김교사로서, 출석 체크가 완료되면 "오늘 출석 체크 완료! 🎉" 같은 재미있는 메시지를 볼 수 있어야 한다. 왜냐하면 작은 성취감을 느끼고 싶기 때문이다.
- [ ] 바쁜 김교사로서, 사이드바에서 "유년부 > 3학년 > 사랑반"처럼 계층적으로 반을 선택할 수 있어야 한다. 왜냐하면 한국 교회의 조직 구조에 맞게 사용하고 싶기 때문이다.

### FEAT-2: 학생/교사 프로필 관리 (MVP)

- [ ] 바쁜 김교사로서, 담당 반 학생들의 기본 정보(이름, 학년, 보호자 연락처)를 조회할 수 있어야 한다. 왜냐하면 심방할 때 연락처를 찾아야 하기 때문이다.
- [ ] 바쁜 김교사로서, 학생의 알레르기나 특이사항을 확인할 수 있어야 한다. 왜냐하면 간식을 준비할 때 참고해야 하기 때문이다.
- [ ] 바쁜 김교사로서, 자신이 담당하는 반 학생만 볼 수 있어야 한다. 왜냐하면 다른 반 학생 정보는 보안상 접근하면 안 되기 때문이다.

### FEAT-3: 심방 기록 및 타임라인 (MVP)

- [ ] 바쁜 김교사로서, 학생별로 심방 이력을 시간순으로 볼 수 있어야 한다. 왜냐하면 이전 심방 내용을 참고하여 연속적인 돌봄을 제공하고 싶기 때문이다.
- [ ] 바쁜 김교사로서, 심방 기록을 작성할 때 유형(전화/심방/카카오톡)을 선택할 수 있어야 한다. 왜냐하면 다양한 방식으로 소통하기 때문이다.
- [ ] 바쁜 김교사로서, 민감한 상담 내용은 비밀 보장 옵션을 선택할 수 있어야 한다. 왜냐하면 담당 교역자만 볼 수 있도록 해야 하기 때문이다.
- [ ] 바쁜 김교사로서, 3주 이상 결석한 학생이 있으면 자동으로 알림을 받을 수 있어야 한다. 왜냐하면 '골든타임'을 놓치지 않고 조기에 개입하고 싶기 때문이다.

### FEAT-4: 게이미피케이션 (v2)

- [ ] 학생로서, 출석하면 달란트 포인트를 받을 수 있어야 한다. 왜냐하면 동기부여가 되기 때문이다.
- [ ] 학생로서, 암송하면 달란트 포인트를 받을 수 있어야 한다. 왜냐하면 말씀 암송을 장려하고 싶기 때문이다.
- [ ] 학생로서, 4주 개근하면 배지를 받을 수 있어야 한다. 왜냐하면 성취감을 느끼고 싶기 때문이다.

## 5. 범위 관리

### 5.1 MVP (반드시 포함)

**Phase 1 - 파일럿 단계:**
- FEAT-0: 기본 온보딩/로그인
- FEAT-1: 스마트 출석 체크 (최우선)
- FEAT-2: 학생/교사 프로필 관리 (기본)
- FEAT-3: 심방 기록 및 타임라인 (기본)

**목표:**
- 출석 체크 시간 50% 단축 (10분 → 5분)
- 교사 만족도 4점 이상 (5점 만점)

### 5.2 옵션 (시간 허용 시)

- QR코드 출석 시스템 (키오스크 모드)
- 엑셀 일괄 업로드 기능

### 5.3 Non-goals (명시적 제외)

**이번 버전에서 하지 않을 것:**
- 게이미피케이션 (달란트, 배지) → v2로 보류
- 통계 리포트 및 대시보드 → v2로 보류
- 학부모 포털 → v3로 보류
- 자동 등반 시스템 → v2로 보류
- SMS/카카오톡 알림 → v2로 보류

**하지 않는 이유:**
- MVP에서는 핵심 기능(출석, 심방)에 집중하여 안정화를 우선시
- 파일럿 단계에서 사용자 피드백을 수집한 후 v2에서 추가

## 7. 해결 가설과 실험

### 7.1 가설

> 만약 교사가 스마트폰으로 **터치 한 번**으로 출석을 체크할 수 있으면, 출석 체크 시간이 **10분에서 3분으로 단축**될 것이다. 왜냐하면 종이에 일일이 작성하는 수작업이 사라지기 때문이다.

### 7.2 실험 루프

| 단계 | 내용 |
|------|------|
| 실험 | 파일럿 단계에서 교사 3~5명에게 FEAT-1(스마트 출석 체크) 사용 |
| 관측 | 출석 체크 전후 시간 측정, 교사 만족도 설문 (5점 척도) |
| 학습 | 출석 체크 시간이 50% 이상 단축되고 만족도 4점 이상이면 성공 |
| 다음 | 성공 시 전체 주일학교로 확장, 실패 시 UI 개선 후 재실험 |
