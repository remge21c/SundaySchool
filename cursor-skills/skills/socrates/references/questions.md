# Socrates 질문 목록 (Q1~Q21) - Cursor 버전

## 질문 진행 방식

**Cursor에서는 일반 대화로 질문합니다.**

각 질문에 예시 선택지를 텍스트로 제공하고, 사용자가 번호/이름/직접 입력으로 답변합니다.

### 질문 형식 예시

```
Q7/21. 꼭 들어가야 할 핵심 기능 3가지를 선택해주세요.

예시 (이 프로젝트에 맞게 조정됨):
1. 식단 기록 - 음식 사진/텍스트로 기록
2. 칼로리 계산 - 자동 영양 분석
3. 주간 리포트 - 통계 및 인사이트
4. 운동 기록 - 운동 시간/종류 추적

3가지를 선택해주세요! (예: 1, 2, 3 또는 직접 입력)
```

Q18 완료 후, AI가 지금까지의 답변을 종합 분석하여 기술 스택을 추천합니다.

| Q | 질문 | AI 추천 방식 |
|---|------|-------------|
| Q19/21 | **백엔드 프레임워크**를 선택해주세요. | AI가 프로젝트 특성 분석 후 추천 (이유 포함) |
| Q20/21 | **프론트엔드 프레임워크**를 선택해주세요. | AI가 사용 환경(Q11) 기반 추천 |
| Q21/21 | **데이터베이스**를 선택해주세요. | AI가 데이터 특성(Q13) 기반 추천 |

### Q19~Q21 진행 방식

1. AI가 Q1~Q18 답변을 종합 분석
2. 프로젝트 특성에 맞는 기술 스택 추천 (이유 포함)
3. 사용자가 AI 추천 수락 또는 직접 선택

### 추천 예시

```
지금까지 답변을 분석한 결과, 당신의 프로젝트에 맞는 기술 스택을 추천드립니다.

---
```

## TDD 기반 개발 안내 (Q21 완료 후)

Q21 완료 후, 다음 안내 메시지를 표시합니다:

```
✅ 기술 스택 선택이 완료되었습니다!

이제 AI 코딩 파트너가 다음 방식으로 개발을 진행합니다:

📋 Contract-First TDD (계약 우선 개발)

1. Phase 0: 계약 & 테스트 설계
   - 모든 API 계약을 먼저 정의
   - 테스트 케이스 선행 작성 (RED 상태)
   - Mock API 생성 (프론트엔드 독립 개발용)

2. Phase 1+: 병렬 구현
   - 백엔드: 테스트 통과 목표로 구현 (RED→GREEN)
   - 프론트엔드: Mock API로 독립 개발 (RED→GREEN)

3. 통합 검증
   - Mock 제거 → 실제 API 연동
   - E2E 테스트로 전체 흐름 검증

이 방식의 장점:
- BE/FE가 서로 기다리지 않고 병렬 개발 가능
- "테스트 통과 = 완료"라는 명확한 기준
- 통합 시 충돌 최소화

이제 6개의 기획 문서를 생성하겠습니다.
```
